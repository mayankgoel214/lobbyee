# Voice worker ↔ app wire contract (Phase 5)

The single source of truth for the HTTP boundary between the Python Pipecat
worker and the Next.js app. The worker holds **no** DB credentials and **no** AI
prompts — it does audio + the guest LLM reply only; everything else is the app's
job. Keep this in sync with:

- `app/api/voice/session-token/route.ts` (mint)
- `app/api/voice/worker/snapshot/route.ts` (read)
- `app/api/voice/worker/turn/route.ts` (write)

## Auth

The browser mints a short-lived token (`POST /api/voice/session-token`, cookie
auth) and hands it to the worker. The worker presents it on every call as:

```
Authorization: Bearer <token>
```

The token is HMAC-signed and carries `{ sessionId, workspaceId, userId, exp }`.
The worker treats it as opaque. All worker endpoints answer:

- `401` — missing / malformed / invalid / expired token (opaque; no body)
- `503` — voice not configured on this deploy (`VOICE_SESSION_TOKEN_SECRET` unset)

## 1. Start of call — `GET /api/voice/worker/snapshot`

No body. Returns everything needed to seed the pipeline:

```jsonc
{
  "sessionId": "uuid",
  "guestSystemPrompt": "You are role-playing a hotel/restaurant GUEST …", // full system instruction, rendered by the app
  "guestSystemVersion": "guest-system@v2",
  "openingCue": "[The session is starting. Greet or approach …]",          // prepend as a user turn if history is empty (the guest speaks first)
  "currentMood": { "frustration": 50, "trust": 50, "patience": 50, "satisfaction": 50 },
  "moodNote": "[Guest mood right now — frustration 50/100, trust 50/100, …]", // rendered; prefix to the NEXT user message
  "history": [ { "role": "user" | "guest", "text": "…" } ]                  // chronological; user/guest only (no coach)
}
```

Errors: `404` (session not visible / not the token's trainee), `409` (session
not `in_progress`).

**Note:** the success criteria (grading rubric) are intentionally absent — the
worker never needs them.

### How the worker builds a guest reply

1. System instruction = `guestSystemPrompt` (verbatim).
2. Seed LLM context from `history`, mapping `guest → model`, `user → user`. If
   `history` is empty (or its first turn is a guest line), prepend `openingCue`
   as the first user turn.
3. For each new staff utterance, the user message sent to the LLM is:
   `` `${moodNote}\n\n${userText}` `` — the mood note is the latest one received
   (from the snapshot, then refreshed by each turn-write response).
4. Stream the reply into TTS. Strip any leading `[…]` the model echoes.

## 2. Each turn — `POST /api/voice/worker/turn`

Sent after the guest has finished speaking. Body:

```jsonc
{
  "idempotencyKey": "uuid",   // one per turn, generated by the worker; reused verbatim on retry
  "userText": "…",            // what the staff member said (STT), 1..4000 chars
  "guestText": "…"            // what the guest replied (and spoke), 1..8000 chars
}
```

The app computes the new mood + coach hint and persists the turn. Response:

```jsonc
{
  "status": "written" | "replayed",
  "guestTurnIndex": 5,
  "mood": { "frustration": 62, "trust": 41, "patience": 38, "satisfaction": 45 },
  "moodNote": "[Guest mood right now — frustration 62/100, …]", // use for the NEXT guest reply
  "coachHint": "Acknowledge the wait before offering a fix." | null
}
```

- `idempotencyKey` makes the write retry-safe: resending the same key replays
  (no duplicate), and a half-written turn is reconciled. **On a network retry,
  reuse the same key.**
- `mood` lags one turn by design: the guest reply you just sent used the
  PREVIOUS mood note; this response gives the mood AFTER this turn, for the next
  reply.

Errors: `400` (bad body), `404` (not the token's session/tenant), `409`
(`status` changed, or a real turn-ordering collision — refetch the snapshot and
retry), `401`/`503` as above.
